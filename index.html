Section 01 · Core Pipeline

投研 Pipeline

八个串联节点，前一步输出自动成为下一步输入。逐字稿 ≥5000 字触发完整流程；BP/简单材料只跑 One-Pager + Question List。点击任意节点查看详情。

01
🚪 Deal Intake
投研入场路由
intake-card.md
+
接收任何形式的材料（BP、链接、描述），自动评估材料质量，判断当前阶段，推荐后续 skill 调用路径。
输入 → BP / 链接 / 描述 聊天记录
独到之处
评估材料质量覆盖度（创始人背景 / 产品 / 数据 / 竞对），标注缺口维度
判断当前所处阶段：初次接触 / 材料积累 / 准备 call / 深度尽调 / IC 准备
自动推荐后续应跑哪几个 skill、以什么顺序运行
在 deals/{公司名}/ 创建项目目录，所有分析产出自动归档
所有项目从这里进入。不跳步骤，不让材料质量差的项目消耗后续 skill 资源。先评估，再行动。
02
📄 Founder One-Pager
公司简介压缩
one-pager.md
+
将所有材料压缩成高密度一页简介：创始人背景、产品、traction、核心差异点、主要风险。
输入 → intake-card.md 原始材料
独到之处
强制区分【事实】【高概率推论】【观点】，不把创始人叙事写成事实
明确列出信息缺口：哪些关键维度还没有证据
涵盖：why now / moat / disconfirming evidence / 主要风险
如果 intake-card.md 已存在，自动读取作为背景，不需要重复粘贴材料
目标是让任何人拿到这页纸，5 分钟内形成有质量的初步判断，而不是读完感觉"说了很多但没有结论"。
03
🗺️ Evidence Map
证据交叉验证
evidence-map.md
+
找出所有"说得通"和"说不通"的地方。哪些有多个来源印证？哪些只有创始人自己说？哪些互相矛盾？
输入 → one-pager.md 多份原始材料
独到之处
区分"多处提到"（可能只是同一个人重复说）与"有独立证据支持"
标出矛盾点：BP 说法 vs 访谈说法 vs 产品实测 vs 外部数据
明确列出：最值得优先验证的 3 个问题
防止被"材料多=信息多"的幻觉带偏
材料越多，越容易产生虚假的置信度。Evidence Map 的作用不是总结材料，而是找出材料之间的裂缝。
04
⚔️ Critical Storyline Attack
反向压力测试
attack.md
+
如果这个项目会死，最可能死在哪？主动找最致命的攻击点，不被漂亮 narrative 带偏。
输入 → evidence-map.md one-pager.md
独到之处
找最致命的一个攻击点，而不是列举所有可能的风险
攻击角度：产品是否有独立存在价值？大模型公司下场会直接颠覆吗？Distribution 被高估了吗？
每个攻击点必须给出具体论据，不输出空洞的"竞争激烈"
默认站在反方立场，比任何 devil's advocate 都更彻底
好的投资判断不只是找理由投，更要找到最硬的理由不投，然后决定是否能反驳它。
05
❓ Diligence Question List
尽调问题清单
question-list.md
+
根据所有已知信息和证据缺口，生成真正值得问的问题。按优先级排序，不问废话。
输入 → attack.md evidence-map.md
独到之处
自动读取所有前序文件，问题直接针对已发现的矛盾点和证据缺口
按优先级分级：P0 必问 / P1 重要 / P2 选问
每个问题注明：想验证什么 / 为什么这个问题信息增量高
不输出模板化客套问题，问题服务于判断质量提升
与 founder 的每一轮 call 时间都是稀缺资源。用错问题是最大的浪费。这个 skill 确保每个问题都在增加信息量。
06
📋 Investment Memo
投资判断备忘录
investment-memo.md
+
综合 Pipeline 所有分析，给出结构化最终判断：thesis · risks · evidence gaps。
输入 → 全量分析文档 founder call 纪要
独到之处
自动读取目录下所有已有文件，综合作为 memo 基础，不让用户重复粘贴
必须包含反方观点（不能只有正面 thesis）
区分：已知事实 / 高概率推论 / 主观判断，结论的确定性层级清晰
给出明确结论：值不值得继续跟进，下一步行动是什么
Memo 不是信息的堆砌，而是判断的文档化。它必须让读者在 10 分钟内知道：这个项目，投还是不投，为什么。
07
🎯 Interview Debrief
访谈复盘
仅模式 A（≥5000字）
+
每次 Founder Call 结束后，AI 自动复盘访谈质量——哪些问题问得好，哪些关键问题遗漏了。
输入 → 逐字稿全文 Diligence Question List
独到之处
引用原文评价：「你问了哪句话」→ 「打开了什么新信息维度」
不是通用建议，而是针对本次 call 的具体缺口分析
给出「下次这个问题可以怎么问」的具体建议
触发条件：逐字稿 ≥5000 字时自动生成；BP/短材料不触发
好的访谈能力是可以训练的。每次 call 完系统给你一份复盘，几十次之后你的提问质量会肉眼可见地提升。
08
✅ Review Checklist
投研自检清单
full-report.md 末尾
+
Pipeline 跑完后，AI 自检一遍所有输出，标出哪些待确认项还没有答案。
检查 → 材料完整性 逻辑一致性 判断质量
独到之处
标出 Evidence Map 中矛盾点是否已处理
标出 One-Pager 与原始材料是否有矛盾
明确下一步行动：是否需要补充尽调、是否进 IC
所有 [ ] 未勾选项 = 下次 founder call 前必须解决的问题
Checklist 不是形式，而是你对自己判断的约束。带着未勾选项去 IC，是对自己和 LP 不负责任。
Section 02 · Intelligence

情报与研究工具

覆盖 AI 人才、市场数据、Builder 动态的实时情报体系。点击卡片查看详情。

🧠
+
AGI Talent Intelligence
AI 人才情报
对 AI/AGI 领域从业者做背景调查、技术主张分析、职业轨迹追踪。生成结构化 profile。
人才 Profile · 背景调查
强制实体消歧：同名同姓的人在 AI 圈很多，先核实是哪个人，再生成 profile
技术主张分析：这个人真正相信什么？跟主流观点的差异在哪？
职业轨迹：从哪里来，往哪里走，转折点是什么
与 candidate-audit 的区别：本 skill 是外部情报（这人是谁），不是招聘判断（要不要招）
情报价值 ≠ 信息量。一份好的人才 profile 应该能告诉你：这个人的判断值不值得认真对待。
📊
+
Market Sizing
市场规模估算
对市场规模、ARPU、留存、渗透率等做合理估算与 sanity check。优先给区间，明确口径。
TAM · ARPU · Sanity Check
优先给区间而不是精确数字，标注高敏感假设（改变一个假设，结论会不会翻转）
区分口径：TAM / SAM / SOM，以及不同口径下的商业逻辑
Sanity check 创始人给的数字：10 亿用户 × 10 美元 ARPU 这种说法成立吗？
从已知参照物往上推：类比 Spotify、LinkedIn 等可验证的数据点
大部分市场规模的问题不是"数字对不对"，而是"用的是哪个口径，这个口径服务于哪个判断"。

+
Follow Builders
AI Builder 动态追踪
每日自动监控 25 位顶级 AI builder 的 X/Twitter 动态 + 5 个主要播客，生成中文摘要推送。
每天 10:00 · 自动推送
25 位 Builder · 5 个播客
内容由中央 feed 统一抓取，无需 API key，名单由 Zara Zhang（YC Partner）维护
覆盖：Karpathy、Sam Altman、Garry Tan、Swyx、Amjad Masad 等核心 builder

Follow builders，not influencers. 直接追踪真正在建造的人，而不是那些转发他们说法的人。
Section 03 · Documents

访谈与文档处理

从转写稿清洗到周报生成，完整覆盖文档生命周期。点击卡片查看详情。

🎙️
+
Transcript
逐字稿整理
将 ASR 自动转写稿（Recall、Otter、飞书妙记等）清洗为可读逐字稿。严格保持原意，不做摘要。
ASR 清洗 · 保持原意
清洗但不改写：去掉语气词、重复、明显的 ASR 错误，但不改变说话人的用词和逻辑
专名、术语、数字：仅在高置信度时纠正，否则保留并标注存疑
按话题分组：让长篇逐字稿有结构，不丢内容
完成后自动提示：要不要继续用 Meeting Consensus 提取共识和行动项
逐字稿的价值在于保留说话人的原始语气和逻辑。过度整理会让你失去最重要的信号。
🤝
+
Meeting Consensus
会议共识提取
从会议纪要、聊天记录、项目复盘中提取共识、分歧与未决问题，以及明确的下一步行动项。
共识 · 分歧 · 行动项
区分：已形成共识 / 存在分歧 / 还没有讨论到的问题
提取好的体验和不好的体验（适用于产品复盘场景）
行动项必须带：负责人 / 截止时间 / 验收标准（能写多少写多少）
可直接衔接 Transcript skill：逐字稿清洗完，一句话触发出纪要
好的纪要不是把会议内容再说一遍，而是让没参会的人5 分钟内知道结论、分歧和下一步。
📅
+
Weekly Review
每周复盘
每周日自动生成结构化周报，含时间分配、工作分类、产出评价、隐性观察、下周建议，及 AI token 统计。
每周日 10:00 · 自动生成
结构化周报 · Token 统计
数据来源：本周日历日程 + 与 AI 的对话记录 + deal-tracker 状态
隐性观察：不是总结做了什么，而是发现时间分配的模式、高/低价值工作的比例
AI 使用统计：本周消耗了多少 token，cache 命中率，估算费用
所有结论区分置信度：高置信事实 / 近似统计 / 数据缺口
周复盘的价值不在于记录发生了什么，而在于发现自己尚未意识到的模式。

AI Wearable · Pre-A · 北美市场
Project NeckLens
颈部可穿戴饮食健康管理设备。通过 Always-on 视觉传感器实时识别饮食行为，结合 AI 算法生成个性化健康建议。创始人 Alex P.，前华为/字节/理想 AI 产品负责人。

01
Founder One-Pager
公司首版结构化理解
+
一句话定位
饮食健康管理领域的 AI 原生可穿戴软硬件，填补「精准行为监测 + 落地建议」的市场空白。
创始人背景
【事实】Alex P.：华为鸿蒙/小艺算法 → 字节 AI 手机/Coze → 理想智能眼镜产品定义，EE 专业 + 视觉控制背景，RoboMaster 全国第一。
强点：产品定义能力待验独立融资经验有限（首次创业）
Why Now
【事实】三个技术窗口同时打开：① 动态低功耗传感器成熟（智能门锁/前摄普及）② Meta SAM 模型发布（1B 掩码，50ms 分割）③ VLM 能力持续提升。
【高概率推论】窗口期约 18-24 个月，先发优势关键。
当前阶段 & Traction
Seed 已完成，产品处于 DVT 阶段（14g 原型机），Q2 计划 Kickstarter 众筹（目标 4000 台，$300-350），Q3 发售。算法准确率当前约 70%。
核心风险（首版）
① 隐私顾虑（颈部摄像头社交接受度）② 算法准确率瓶颈 ③ 硬件 ID 与目标客群匹配度 ④ 北美市场冷启动路径验证不足
02
Evidence Map
Claim 独立性分析 + 证据交叉验证
+
Claim #1 · 饮食识别准确率 70%
同源复述 来源：仅创始人口述，无第三方测试数据
置信度：低 ｜ 反证：复杂菜肴（印度菜/混合食物）创始人自认"完全做不准"
→ 验证路径：要求提供内部测试数据集；或安排产品实测
Claim #2 · 美国成人慢性病市场 2 亿人
强独立 来源：CDC 数据 + 创始人引用 + MyFitnessPal 用户规模（几亿注册）
置信度：高 ｜ 需求真实，但支付意愿和硬件接受度仍需验证
⚠️ 矛盾点
创始人称"北美用户 90% 喜欢黄金颜色"，但同时称目标用户是"35-55岁有慢性病风险人群"——这两个说法在审美偏好上明显矛盾，前者来自小样本广告测试，后者是产品设计主轴。
外观设计定位与目标用户群之间的张力尚未解决。
03
Critical Storyline Attack
最致命攻击点
+
🔴 核心攻击点：「颈部摄像头」是产品的最大障碍，不是功能，是社交接受度
创始人已承认：用户在公共场合最大的顾虑是"让别人摘掉"而非隐私政策。四十多个用户调研中，隐私顾虑是首要反馈。

为什么这是最致命的：饮食场景天然是社交场景（餐厅、家庭聚餐），这恰好是产品需要工作的核心场景，同时也是社交压力最大的场景。产品需要最准确工作的时候，用户最可能摘下它。

如果攻击成立：Kickstarter 的前 4000 台可能被科技爱好者购买，但无法突破到主流用户（35-55 岁慢性病人群），核心假设"从科技圈到慢性病人群"的破圈路径断裂。

创始人的回应：镜头短焦（30-50cm 后变糊）+ 端上人脸模糊。这解决了隐私技术问题，但没有解决社交观感问题。
04
Investment Memo · 结论节
当前阶段判断
+
当前结论
🟡 Pending — 值得继续跟，但核心假设未验证，不建议当前阶段推进 IC
支撑 thesis 的理由
① 需求真实：软件解法长期无法解决行为流数据缺失问题，硬件切入有逻辑
② Why Now 成立：三个技术窗口同时打开，时机判断合理
③ 创始人产品能力强：从概念到 DVT 仅 4 个月，执行力有迹可循
核心悬而未决
① 社交接受度能否支撑「科技圈 → 慢性病人群」破圈（最关键，目前无数据）
② 算法准确率能否从 70% 提升到用户可接受阈值（80%+）
③ Kickstarter 是否能验证目标用户画像（vs. 只是科技爱好者）
建议下一步
等 Kickstarter 结果（Q2）；重点关注：购买者画像是否含慢性病人群、1个月留存率、产品实测准确率数据
